Create the following directory tree under `templates/registry/`:

**templates/registry/worker/src/index.ts** -- Adapted from Lens Worker with these changes:
- Service name: `prism-registry` (not `lens-worker`)
- User-Agent: `Prism-Worker/1.0` (not `Lens-Worker`)
- Routes:
  - `GET /registry` AND `GET /api/skills/registry` -- both return skill-registry.json from GitHub (alias for backward compat)
  - `GET /api/skills/:name` -- return specific skill's SKILL.md from GitHub
  - `POST /api/skills/publish` -- accept Prism payload format (flat fields), reconstruct plugin.json + SKILL.md, create branch + PR via GitHub Git API
  - `GET /health` -- return `{"status": "ok", "service": "prism-registry"}`
- Auth: Bearer token validated against `REGISTRY_TOKENS` env secret (comma-separated list). `const tokens = env.REGISTRY_TOKENS.split(",").map(t => t.trim()); if (!tokens.includes(bearerToken)) return unauthorized;`
- Env interface: `interface Env { GH_OWNER: string; GH_REPO: string; GH_BRANCH: string; GH_TOKEN: string; REGISTRY_TOKENS: string; }`
- Publish validation (`validatePrismPublish`): Each skill must have `name` (kebab-case regex `^[a-z0-9][a-z0-9-]*[a-z0-9]$`), `content` (string, min 50 chars), `repository` (non-empty string), `description` (non-empty string), `author` (non-empty string). All skills must share same `repository`.
- File building: For each skill, create `skills/{repository}/{name}/plugin.json` (reconstructed from flat fields: name, description, author, repository, category, source, commit_date, source_hash) and `skills/{repository}/{name}/SKILL.md` (from `content` field).
- PR creation: Same Git API pattern as Lens (get base ref, create blobs, create tree, create commit, create branch, create PR). Branch name: `prism/publish-{timestamp}`. PR title: `"Publish skills from {repository}"`.
- CORS: Allow `*` origin for GET requests. No CORS on POST (server-to-server only).
- Error responses: JSON `{"error": "message"}` with appropriate status codes.
- GitHub fetch helper: `fetchFromGitHub(env, path)` fetching `https://api.github.com/repos/{owner}/{repo}/contents/{path}?ref={branch}` with headers `Authorization: token {GH_TOKEN}`, `Accept: application/vnd.github.v3.raw`, `User-Agent: Prism-Worker/1.0`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Prism client -> Worker API | Skills published over HTTPS with Bearer token auth |
| Worker -> GitHub API | Worker proxies to GitHub using GH_TOKEN secret |
| GitHub PR -> main branch | Human review gate before skills merge to registry |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Spoofing | Worker auth | mitigate | Validate Bearer token against REGISTRY_TOKENS comma-separated secret. Return 401 for missing/invalid tokens. |
| T-04-06 | Tampering | Skill content injection | mitigate | Worker validates skill name regex `^[a-z0-9][a-z0-9-]*[a-z0-9]$`, content min 50 chars, required fields non-empty. CI runs validate.py on PRs. Human review on merge. |
| T-04-07 | Information Disclosure | GH_TOKEN exposure | mitigate | GH_TOKEN stored as Wrangler secret (encrypted at rest), never in wrangler.toml vars. Worker never returns it in responses. |
| T-04-08 | Denial of Service | Large publish payloads | mitigate | Cloudflare Workers have 100MB request size limit by default. Worker validates array length and individual skill content length before processing. |
| T-04-09 | Elevation of Privilege | Worker-side code execution | accept | Cloudflare Workers V8 isolate model prevents filesystem/network access beyond fetch(). No eval() in Worker code. Low risk. |
</threat_model>
